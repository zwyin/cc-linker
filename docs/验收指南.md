# cc-bridge 验收指南

本文档从实际使用场景出发，详细说明如何验收 cc-bridge 的各项功能。

## 一、环境准备

### 1.1 前置条件

| 依赖 | 版本要求 | 检查命令 | 说明 |
|------|---------|---------|------|
| Bun | >= 1.0 | `bun --version` | 运行时和包管理器 |
| Claude Code | 最新版 | `claude --version` | CLI 工具 |
| cc-connect | 可选 | `cc-connect --version` | 飞书桥接（如需飞书集成） |

### 1.2 目录结构

验收前确认以下目录存在：

```bash
# Claude Code 会话目录
ls ~/.claude/projects/

# cc-connect 会话目录（如已安装）
ls ~/.cc-connect/sessions/

# cc-bridge 数据目录（初始化后创建）
ls ~/.cc-bridge/
```

## 二、安装验收

### 2.1 从源码安装

```bash
# 步骤 1：克隆仓库
git clone https://github.com/xxx/cc-bridge.git
cd cc-bridge

# 步骤 2：安装依赖
bun install
# 预期：无报错，node_modules 目录创建成功

# 步骤 3：类型检查
bun run typecheck
# 预期：无类型错误

# 步骤 4：运行测试
bun test
# 预期：所有测试通过

# 步骤 5：编译
bun run build
# 预期：dist/cc-bridge 文件生成

# 步骤 6：验证编译产物
./dist/cc-bridge --help
# 预期：显示帮助信息，包含所有命令列表

# 步骤 7：加入 PATH
export PATH="$(pwd)/dist:$PATH"
cc-bridge --help
# 预期：同上
```

**验收标准**：
- [ ] `bun install` 无报错
- [ ] `bun run typecheck` 无类型错误
- [ ] `bun test` 所有测试通过
- [ ] `bun run build` 生成 `dist/cc-bridge`
- [ ] `cc-bridge --help` 显示完整帮助信息

## 三、初始化验收

### 3.1 首次初始化

```bash
# 步骤 1：运行初始化
cc-bridge init

# 预期输出：
# ✅ Created ~/.cc-bridge/registry.json
# 🔍 Scanning for existing sessions...
#    Found X cc-connect sessions
#    Found Y Claude Code sessions
# ✅ Registered Z sessions total
#
# Next steps:
#   1. Run 'cc-bridge hook install' to install Claude Code hook
#   2. Run 'cc-bridge list' to view all sessions
#   3. Run 'cc-bridge resume' to resume a session

# 步骤 2：验证目录结构
ls -la ~/.cc-bridge/
# 预期：包含以下文件/目录
# - registry.json      # 会话注册表
# - backups/           # 备份目录
# - scan_cache.json    # 扫描缓存

# 步骤 3：验证 registry 内容
cat ~/.cc-bridge/registry.json | jq .
# 预期：JSON 格式，包含 version、updated_at、sessions 字段
```

**验收标准**：
- [ ] `~/.cc-bridge/registry.json` 文件创建成功
- [ ] `~/.cc-bridge/backups/` 目录创建成功
- [ ] `~/.cc-bridge/scan_cache.json` 文件创建成功
- [ ] registry.json 格式正确，包含已扫描的会话

### 3.2 重复初始化

```bash
# 步骤 1：再次运行初始化
cc-bridge init

# 预期输出：
# 📁 Registry exists at ~/.cc-bridge/registry.json, will refresh
# 🔍 Scanning for existing sessions...
#    ...

# 步骤 2：验证数据完整性
cc-bridge list
# 预期：显示会话列表，数据未丢失
```

**验收标准**：
- [ ] 重复初始化不会丢失已有数据
- [ ] 会话数量保持一致或增加（如有新会话）

## 四、Hook 安装验收

```bash
# 步骤 1：安装 Hook

# 预期输出：
# Hook 安装成功
# 已添加到 ~/.claude/settings.json:
#   "hooks": { "SessionStart": "cc-bridge hook session-start" }

cat ~/.claude/settings.json | jq .hooks
# 预期：
# {
#   "SessionStart": "cc-bridge hook session-start"
# }

# 步骤 3：检查 Hook 状态
cc-bridge hook status
# 预期：
# Hook 状态: 已安装
```

**验收标准**：
- [ ] `~/.claude/settings.json` 中包含 Hook 配置
- [ ] `cc-bridge hook status` 显示"已安装"

### 4.2 Hook 功能验证

```bash
# 步骤 1：启动新的 Claude Code 会话
cd /tmp && claude

# 步骤 2：在会话中做一些操作
# （输入一些内容，等待响应）

# 步骤 3：退出会话

# 步骤 4：检查 registry 是否自动注册

# 步骤 5：查看 Hook 日志

# 预期：包含 "已注册会话 xxx (origin: cli, cwd: /tmp)" 类似日志
```

**验收标准**：
- [ ] 新启动的 Claude Code 会话自动注册到 registry
- [ ] Hook 日志记录注册成功

## 五、基本命令验收

### 5.1 `cc-bridge list` - 列出会话

```bash
# 场景 1：默认列表
cc-bridge list
# 预期：表格形式显示会话，包含 Ref、标题、来源、项目、消息数、最后活跃时间

# 场景 2：按来源过滤
cc-bridge list --origin cli
# 预期：只显示终端发起的会话

cc-bridge list --origin cc-connect
# 预期：只显示 cc-connect 发起的会话

# 场景 3：按项目过滤
cc-bridge list --project cc-bridge
# 预期：只显示项目名包含 "cc-bridge" 的会话

# 场景 4：显示最近活跃
cc-bridge list --active
# 预期：只显示最近 2 小时内活跃的会话

# 场景 5：限制数量
cc-bridge list --limit 5
# 预期：最多显示 5 个会话

# 场景 6：JSON 格式输出
cc-bridge list --format json
# 预期：输出 JSON 格式的会话列表

# 场景 7：CSV 格式输出
cc-bridge list --format csv
# 预期：输出 CSV 格式的会话列表

# 场景 8：显示已归档会话
cc-bridge list --archived
# 预期：显示 status 为 archived 或 corrupted 的会话
```

**验收标准**：
- [ ] 默认表格格式正确显示
- [ ] 各过滤选项（--origin、--project、--active）正常工作
- [ ] --limit 限制数量正确
- [ ] --format json/csv 输出格式正确
- [ ] --archived 显示归档会话

### 5.2 `cc-bridge resume` - 恢复会话

```bash
# 场景 1：通过 UUID 前缀恢复
cc-bridge resume b21d6d04
# 预期：
# - 提示恢复会话标题
# - 切换到会话的工作目录
# - 启动 claude --resume <uuid>

# 场景 2：交互式选择
cc-bridge resume
# 预期：
# - 显示会话列表供选择
# - 选择后恢复该会话

# 场景 3：搜索后恢复
cc-bridge resume --search "认证"
# 预期：
# - 搜索标题包含 "认证" 的会话
# - 如果只有一个匹配，直接恢复
# - 如果多个匹配，显示选择列表

# 场景 4：恢复最近会话
cc-bridge resume --latest
# 预期：恢复最近活跃的会话

# 场景 5：Dry Run 模式
cc-bridge resume b21d6d04 --dry-run
# 预期：只显示将执行的命令，不实际执行
# 输出：将执行: cd /path/to/project && claude --resume <uuid>

# 场景 6：指定工作目录
cc-bridge resume b21d6d04 --cwd /new/path
# 预期：在指定目录下恢复会话

# 场景 7：跳过确认
cc-bridge resume b21d6d04 --no-confirm
# 预期：不提示确认，直接恢复
```

**验收标准**：
- [ ] UUID 前缀匹配正确
- [ ] 交互式选择列表正常显示
- [ ] --search 搜索功能正常
- [ ] --latest 恢复最近会话
- [ ] --dry-run 只显示命令不执行
- [ ] --cwd 指定目录正确
- [ ] 工作目录切换正确

### 5.4 `cc-bridge sync` - 同步会话

```bash

#    New sessions registered: Z
#    Sessions updated: W

# 🔄 Syncing sessions...

# 场景 3：强制刷新

cc-bridge sync --clean

# 预期输出：

#    Cleaned N invalid sessions
#    ...
```

**验收标准**：
- [ ] 同步后会话数量正确
- [ ] --scan 不写入 registry
- [ ] --force 强制全量扫描
- [ ] --clean 清理无效记录

## 七、配置验收

### 7.1 配置文件

```bash
# 步骤 1：创建配置文件
cat > ~/.cc-bridge/config.toml << 'EOF'
[general]
log_level = "debug"
log_path = "~/.cc-bridge/cc-bridge.log"

[bridge]
api_url = "http://localhost:9810"
token = "my-secret-token"

[scanner]
max_file_size = 52428800  # 50MB
EOF

# 步骤 2：验证配置生效
cc-bridge status
# 预期：使用配置文件中的设置

# 步骤 3：查看日志
cat ~/.cc-bridge/cc-bridge.log
# 预期：包含 debug 级别日志
```

**验收标准**：

## 十二、常见问题

### Q1：为什么 `cc-bridge list` 没有显示任何会话？

**可能原因**：
1. 未运行 `cc-bridge init`
2. cc-connect 和 Claude Code 都未创建过会话
3. 会话目录路径不正确

**解决方法**：
```bash
# 1. 运行初始化
cc-bridge init

# 2. 检查会话目录
ls ~/.cc-connect/sessions/
ls ~/.claude/projects/

# 3. 手动同步
cc-bridge sync --force
```

### Q2：为什么 Hook 安装后新会话没有自动注册？

**可能原因**：
1. Hook 未正确安装
2. Claude Code 版本不支持 Hook
3. Hook 执行失败

**解决方法**：
```bash
# 1. 检查 Hook 状态
cc-bridge hook status

# 2. 查看 Hook 日志
cat ~/.cc-bridge/hook.log

# 3. 重新安装 Hook
cc-bridge hook uninstall
cc-bridge hook install
```

### Q3：为什么 `cc-bridge resume` 提示 JSONL 文件不存在？

**可能原因**：
1. 会话的 JSONL 文件被删除
2. 会话目录路径变化

**解决方法**：
```bash
# 1. 检查会话状态
cc-bridge show <uuid>

# 2. 标记为 corrupted
cc-bridge clean

# 3. 如果文件确实不存在，无法恢复
```

### Q4：为什么 `/bridge switch` 需要重启 cc-connect？

**原因**：
cc-connect 当前仅在启动时加载 session JSON，运行中不监听文件变化。首次映射 CLI 会话需要写入 session JSON 并重启 cc-connect 使其生效。

**解决方法**：
1. 使用 `--confirm` 参数确认重启
2. 或等待 cc-connect 支持热加载（未来版本）

### Q5：如何备份和恢复 registry？

**备份**：
```bash
# 自动备份在 ~/.cc-bridge/backups/ 目录
ls ~/.cc-bridge/backups/

# 手动备份
cp ~/.cc-bridge/registry.json ~/registry-backup.json
```

**恢复**：
```bash
# 从备份恢复
cp ~/.cc-bridge/backups/registry.20260503_103000.json ~/.cc-bridge/registry.json

# 或重新初始化
cc-bridge init
```
